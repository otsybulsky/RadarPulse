# Розділ 4: Монастир Домену

Уявіть собі старовинний, обнесений високими кам'яними стінами монастир, що стоїть на вершині неприступної скелі. Всередині цього монастиря ченці зберігають священні сувої з законами природи. Вони не цікавляться політикою сусідніх держав, курсом валют чи новими модами в одязі. Їхнє життя підпорядковане вічним, незмінним правилам. Що б не відбувалося у зовнішньому світі — зміна королів, війни чи наукові відкриття — ченці продовжують переписувати свої сувої точно так само, як і сто років тому.

В архітектурі системи **RadarPulse** роль такого ізольованого монастиря відіграє **Domain Layer** (Доменний шар).

## 4.1. Закон чистого домену (Domain Layer)

Більшість сучасних програмних проектів страждають від однієї хронічної хвороби: вони «забруднюють» свою бізнес-логіку технологіями.

Ви відкриваєте файл, де має розраховуватися фізичний стан грозового фронту, і бачите там SQL-запити до бази даних PostgreSQL, анотації для серіалізації в JSON, виклики логера Serilog або прямі звернення до черги Kafka. Це архітектурна катастрофа. Якщо завтра ви вирішите замінити SQL-базу на сховище NoSQL або перенести систему з Kafka на RabbitMQ, вам доведеться переписати саму суть вашої програми — її математику та логіку обробки. Детектив у такому коді швидко втратить слід, бо бізнес-правила будуть поховані під тоннами інфраструктурного сміття.

У RadarPulse встановлено суворий архітектурний режим: **Домен (`RadarPulse.Domain`) тримається чистим від інфраструктурних залежностей.**

Він не має жодної залежності від зовнішніх шарів. Він нічого не знає про:
* Існування баз даних, файлових систем чи мережевих стеків.
* Веб-контролери, HTTP-протокол, CORS чи JSON-серіалізатори.
* Хмарні сервіси, контейнери Kubernetes чи черги повідомлень.
* Навіть про фреймворки для модульних тестів (домен тестується ззовні).

Доменний шар володіє лише чистими правилами обробки радарів, математичними формулами розрахунку дельт, логікою групування подій та політиками динамічного ребалансу шардів. Якщо завтра зміниться спосіб доставки — CLI, BFF, файл, база чи broker adapter — core-правила не повинні переписуватися разом із ним. Їхній основний партнер — базові типи C# та доменні контракти, а не HTTP, SQL чи конкретний формат UI.

## 4.2. Шар Application: Приймальня монастиря

Але як тоді монастирю спілкуватися із зовнішнім світом? Як передати радарні дані всередину системи та зберегти результати обробки на диск?

Для цього існує шар **Application** (`RadarPulse.Application`). Він виступає в ролі приймальні нашого монастиря — місця, куди подорожні можуть приносити свої прохання та отримувати відповіді, але не мають права заходити в келії ченців.

Шар Application визначає дві ключові речі:
1. **Порти (Ports):** Це абстрактні інтерфейси C#, які описують, що саме доменному монастирю потрібно від зовнішнього світу для виконання своєї роботи.
2. **Use Cases (Сценарії використання):** Це координаційні сценарії, які беруть вхідні дані, проганяють їх крізь доменні правила і передають результат далі.

Наприклад, монастирю потрібно зафіксувати історію обробки батча. Домен не знає, як писати файли на диск. Тому в шарі Application ми оголошуємо інтерфейс-порт:

```csharp
public interface IRadarPulseProductPipelineHistoryService
{
    Task<IReadOnlyCollection<RadarPulseProductRunRecord>> GetRunHistoryAsync();
    Task RecordRunAsync(RadarPulseProductRunRecord record);
    Task ClearHistoryAsync();
}
```

Це залізна декларація намірів. Шар Application каже: «Мені байдуже, як саме ви реалізуєте це збереження — запишете у хмару Azure, збережете у файл `radarpulse-product-history.json` чи просто намалюєте олівцем на стіні бункера. Головне, щоб ви надали мені ці методи».

Реалізація цього порту — це завдання зовнішнього шару **Infrastructure** (`RadarPulse.Infrastructure`), який лежить далеко за межами нашого монастиря.

## 4.3. POCO та захист від технологічної голки (Vendor Lock-in)

Звідки береться технологічна залежність? Вона підкрадається непомітно. Розробник вирішує: «Мені просто потрібно, щоб цей доменний об'єкт серіалізувався в JSON для відправки клієнту. Я просто додам атрибут `[JsonPropertyName("volume_id")]` над полем». Або: «Нам потрібно зберігати цю сутність у базі даних, я просто додам анотацію `[Table("radar_runs")]` та `[Key]`».

В цей момент чистоту домену втрачено. Домен «підсаджується на технологічну голку» конкретного ORM-фреймворку (наприклад, Entity Framework) або бібліотеки серіалізації. Якщо ці бібліотеки оновляться або зміняться, бізнес-правила системи постраждають.

Щоб запобігти цьому, всі сутності в `RadarPulse.Domain` є чистими **POCO** (Plain Old CLR Objects) та некерованими структурами:
* **Жодних сторонніх атрибутів:** Поля сутностей не містять розмітки для баз даних чи серіалізаторів.
* **Мапування даних (Data Mapping):** Якщо інфраструктурі потрібно серіалізувати батч або зберегти метрики в файл, вона створює свої власні інфраструктурні DTO-моделі (Data Transfer Objects) та самостійно перекладає чисті доменні об'єкти в потрібний формат.
* **Незалежність компіляції:** Проект `Domain` не посилається на жодні зовнішні NuGet-пакети. Його залежності — це лише ядро .NET CLR.

Це дає нам повну свободу дій. Ми можемо повністю переписати BFF-контролери чи замінити JSON-серіалізатор на бінарний MessagePack, не змінивши при цьому жодного рядка коду в нашому ядрі обробки радарів.


## 4.4. Напрямок залежностей: Односторонній рух

Головне правило Clean Architecture, яке ми оберігаємо як зіницю ока — це **напрямок залежностей**. Рух дозволено тільки в один бік: ззовні до центру.

```
┌─────────────────────────────────────────────────────────────┐
│ PRESENTATION Layer (HTTP API / CLI / Angular UI Host)       │
└──────────────────────────────┬──────────────────────────────┘
                               │ (посилається на)
                               ▼
┌─────────────────────────────────────────────────────────────┐
│ INFRASTRUCTURE Layer (File Stores / DB Adapters / Workers)  │
└──────────────────────────────┬──────────────────────────────┘
                               │ (реалізує порти та посилається на)
                               ▼
┌─────────────────────────────────────────────────────────────┐
│ APPLICATION Layer (Ports / Use Cases / Product DTOs)        │
└──────────────────────────────┬──────────────────────────────┘
                               │ (посилається на)
                               ▼
┌─────────────────────────────────────────────────────────────┐
│ DOMAIN Layer (Pure Entities / Invariants / Validators)      │
└─────────────────────────────────────────────────────────────┘
```

* **Domain** стоїть у центрі всесвіту. Він не посилається ні на що.
* **Application** посилається тільки на `Domain`.
* **Infrastructure** посилається на `Application` та `Domain`. Вона реалізує інтерфейси-порти, визначені в Application.
* **Presentation** (HTTP-контролери, консольні команди CLI) посилається на `Application` та `Domain`, запускаючи потрібні сценарії використання.

Така структура робить серце системи ізольованим для рев'ю й тестування. Детектив, який розслідує помилку в розрахунках кутів відбиття радарного променя, починає в межах `RadarPulse.Domain`, не відволікаючись на код підключення до веб-сервера чи логіку CORS-політики.

Але як гарантувати, що в процесі щоденної розробки, під час поспіху перед демо-показом або при рефакторингу якийсь втомлений програміст випадково не порушить ці священні кордони? Як захистити монастир від набігу інфраструктурних кочівників? Для цього ми виставили на стінах озброєних вартових, про яких піде мова в наступному розділі.

## 🔍 Матеріали справи (Investigation Case Files)

### 1. Вердикт детективів (Decision Trace & Rationale)
Суворе дотримання кордонів Clean Architecture (Milestone `036`). Проект `Domain` є ізольованим монастирем без будь-яких зовнішніх посилань (наприклад, на бази даних або сервіси логування). Усі інфраструктурні дії реалізовані як адаптери в `Infrastructure`, які задовольняють контракти use-case портів з `Application`.

#### Чому домен лишився без зовнішніх ключів
Найшвидший шлях у короткій перспективі — дозволити домену знати про файли, HTTP, логування й конкретні DTO. Так код спершу здається зручнішим, але кожен такий shortcut лишає на місці злочину чужі відбитки: домен починає залежати від деталей доставки. Інший шлях — зробити «чистоту» декларацією без автоматичного контролю; це тримається рівно до першого поспіху. Ми обрали жорсткий кордон шарів, бо core-алгоритми мають пережити зміну CLI, BFF, durable store і UI. Ціна вибору — більше портів і мапінгу; виграш — домен можна перевіряти як чистий механізм.

### 2. Закони фізики рантайму (System Invariants)
* **Напрямок залежностей**: Жоден клас із `Domain` або `Application` не може посилатися на `Infrastructure`.
* **Чистота домену**: Заборонено використання будь-яких інфраструктурних атрибутів (наприклад, JSON.NET анотацій) всередині доменних сутностей.

### 3. Патологоанатомічний звіт (Failure Modes & Recovery)
* **Архітектурні порушення**: Спроба підключити інфраструктурну бібліотеку безпосередньо до домену виявляється на стадії компіляції або блокується статичними аналізаторами коду.
