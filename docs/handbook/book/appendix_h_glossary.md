# Додаток 8: Глосарій RadarPulse і .NET-термінів

Цей глосарій зібраний для читача, який хоче швидко вирівняти мову книги: що є терміном RadarPulse, що є загальним .NET/C# поняттям, а що лишається англомовною назвою через API, бібліотеку або усталену інженерну практику.

Глосарій не замінює розділи книги. Він дає коротке визначення, щоб під час читання не доводилося щоразу повертатися до першого місця, де термін пояснювався.

## Дані, архів і погодні радари

| Термін | Просте пояснення |
| :--- | :--- |
| **NEXRAD** | Американська мережа погодних радарів. У книзі це головне джерело реальних бінарних даних для RadarPulse. |
| **NEXRAD Level II** | Формат сиріших радарних даних з об'ємними сканами, моментами вимірювання і службовими заголовками. Це формат даних, а не “розширення файла”. |
| **NOAA** | Національна океанічна й атмосферна адміністрація США, яка публікує погодні архіви. |
| **AWS Open Data** | Публічні набори даних на AWS. У книзі це маршрут доступу до відкритого архіву NEXRAD без приватного джерела автора. |
| **S3 bucket** | Об'єктне сховище AWS. NEXRAD-архів доступний як публічний bucket, з якого RadarPulse може будувати локальний кеш. |
| **KTLX** | Ідентифікатор конкретного NEXRAD-радара, який використовується в книзі як частина локального корпусу/прикладів. |
| **`radarId`** | Ідентифікатор радара в шляхах кешу, маніфестах або командах завантаження. |
| **Радарний архів** | Набір історичних NEXRAD-файлів, які можна завантажити, кешувати локально і проганяти через систему повторно. |
| **Volume-файл** | Один файл радарного об'єму: серія сканів, зібрана радаром за певний проміжок часу. |
| **Archive Two** | Контейнерна структура NEXRAD Level II з записами, заголовками і стисненими блоками. |
| **BZip2** | Алгоритм стиснення, який часто використовується в погодних архівах через добрий баланс між розміром і можливістю зберігати великі історичні масиви. |
| **BZip2-блок** | Окрема стиснена частина даних, яку можна декомпресувати як одиницю роботи. |
| **Move-To-Front** | Крок у BZip2, який замінює повторювані символи позиціями в рухомому списку, щоб подальше кодування краще стискало дані. |
| **Дерева Гаффмана** | Структури кодування, які BZip2 використовує на етапі стиснення/розпакування. У книзі вони важливі як частина ціни повторної ініціалізації декомпресора. |
| **`BZip2InputStream`** | Потокова обгортка з поширених бібліотек декомпресії. У книзі згадується як приклад зручного, але алокаційно дорогого шляху для гарячого архівного контуру. |
| **Маніфест файлу** | Карта вмісту архівного файла: де починаються записи, які блоки стиснені і що треба читати без сліпої декомпресії всього підряд. |
| **Random access** | Доступ не обов'язково з початку до кінця файла, а через відомі зміщення. У RadarPulse це залежить від попередньо збудованої карти/маніфесту. |
| **Payload** | Корисне навантаження: масив значень, який реально потрібен для обчислень, на відміну від службових заголовків. |
| **Payload value** | Одне числове значення в корисному навантаженні. Саме в цих одиницях часто вимірюється пропускна здатність. |
| **Gate** | Комірка вимірювання вздовж променя радара: умовний “крок відстані” від радара. |
| **Moment** | Тип радарного вимірювання, наприклад відбиття, швидкість або інший фізичний показник. |
| **Gate-moment value** | Значення конкретного моменту в конкретному gate. Це одна з природних дрібних одиниць радарного потоку. |
| **Source / `sourceId`** | Джерело подій або група даних, яку RadarPulse маршрутизує, шардить і відстежує окремо. |
| **Checksum** | Контрольна сума, що допомагає перевірити, чи дані не змінилися між етапами обробки. |

## Архітектура RadarPulse

| Термін | Просте пояснення |
| :--- | :--- |
| **RadarPulse** | Інженерний стенд і система обробки радарних даних, побудована як доказовий проект: код, тести, вимірювання і межі тверджень. |
| **Lab-table стенд** | Лабораторний режим системи: локальні дані, повторювані сценарії, контрольовані вимірювання. Це не production-контур. |
| **Clean Architecture** | Архітектурний підхід, де домен не залежить від інфраструктури, UI, баз даних або логерів. |
| **Domain** | Внутрішнє ядро правил і моделей. У RadarPulse саме тут мають жити інваріанти, контракти і чисті доменні рішення. |
| **Application** | Шар сценаріїв використання: координує домен, будує read models і готує дані для продуктового шару. |
| **Infrastructure** | Адаптери до файлів, архівів, HTTP, декомпресії, history store та інших зовнішніх механізмів. |
| **Presentation** | Зовнішня поверхня: CLI, HTTP API, BFF і UI. |
| **POCO** | Plain Old CLR Object: простий .NET-об'єкт без залежності від фреймворку, бази даних або магічного runtime-контексту. |
| **Контракт домену** | Явне правило, яке об'єкт має виконати перед тим, як потрапити в ядро або перейти на наступний етап. |
| **Guard-перевірка** | Швидка перевірка на вході, яка зупиняє некоректний стан до того, як він розповзеться системою. |
| **Інваріант** | Умова, яка завжди має лишатися правдивою: порядок, межі пам'яті, валідність батча, цілісність payload. |
| **Fail-closed** | Стратегія безпеки: якщо система не може довести коректність, вона зупиняє прийом або блокує дію, а не продовжує з неправдою. |
| **Non-claim** | Те, що система чесно не заявляє як готове. Наприклад: production logging, multi-node consensus або live ingestion можуть бути напрямами розвитку, але не поточним доказом. |
| **Scope limit** | Межа відповідальності твердження: на якому корпусі, залізі, режимі й версії коду воно доведене. |
| **Production boundary** | Межа між доведеним лабораторним стендом і реальною продукційною експлуатацією. |
| **Production hardening** | Набір робіт, які переводять lab-table інваріанти в production: security, broker/database adapters, observability, live ingestion, deployment gates. |
| **IAM** | Identity and Access Management: автентифікація, авторизація, ролі й доступи. У книзі це production-hardening тема, а не частина поточного лабораторного ядра. |
| **AuthN/AuthZ** | AuthN означає автентифікацію: хто ти. AuthZ означає авторизацію: що тобі дозволено робити. |
| **Fencing** | Захисний маркер або механізм, який не дозволяє старому власнику ресурсу продовжити дію після втрати права на неї. |
| **NetArchTest** | .NET-бібліотека для архітектурних тестів. У RadarPulse вона описана як відомий підхід, але фактичні архітектурні тести реалізовані вручну. |

## Потік, батчі та паралельний runtime

| Термін | Просте пояснення |
| :--- | :--- |
| **Runtime / рантайм** | Частина системи, яка виконує обробку: приймає батчі, запускає воркери, координує порядок і фіксацію. |
| **Hot path / гарячий шлях** | Найчастіше виконуваний шлях коду, де навіть дрібна алокація або зайвий виклик можуть стати дорогими. |
| **Cold start / холодний старт** | Перший запуск або перша обробка, коли пул, JIT і кеші ще не прогріті. |
| **Steady-state** | Стабільний режим після прогріву, коли система вже не платить стартові витрати. |
| **Warm cache / прогрітий кеш** | Стан, коли дані або структури вже були прочитані/підготовлені, тому наступний прогін швидший. |
| **Worker / воркер** | Паралельний виконавець, який бере частину роботи й рахує її незалежно від інших. |
| **Batch / батч** | Пакет подій, який система обробляє як одну одиницю маршрутизації, обчислення і фіксації. |
| **Batch boundary** | Межа батча. У RadarPulse багато безпечних рішень, наприклад міграція або commit, дозволені саме на таких межах. |
| **`RadarStreamEvent`** | Компактна структура події гарячого шляху. Вона замінює розсип дрібних об'єктів. |
| **`RadarEventBatch`** | Доменний пакет подій і payload-діапазонів, який передається в обробку. |
| **Provider sequence** | Монотонний номер батча від провайдера. Він задає порядок, у якому результати мають бути зафіксовані. |
| **Ordered commit / впорядкована фіксація** | Правило: рахувати можна паралельно, але фінальний стан змінюється тільки в правильному порядку sequence. |
| **Compute phase / фаза обчислення** | Етап, де воркери паралельно рахують результат або дельту. |
| **Commit phase / фаза фіксації** | Етап, де система застосовує готовий результат до фінального стану. |
| **Delta / дельта** | Немутабельний звіт про зміни, які батч додав до стану. Дельта не змінює ядро сама. |
| **Moving average delta** | Дельта для ковзного середнього: має не лише додану суму, а й інформацію про те, що входить і виходить з вікна. |
| **Merge / злиття** | Об'єднання дельт у фінальний стан за визначеним правилом. |
| **Mergeable** | Властивість обробника: його проміжні результати можна безпечно зливати. |
| **Sequential fallback** | Резервний послідовний шлях, коли паралельне злиття не доведене або заборонене. |
| **SnapshotOnly** | Класифікація обробника, результат якого можна читати зі знімка, але не можна запускати через ordered delta/merge шлях. |
| **Unsupported** | Класифікація, яка блокує набір обробників для MVP-поверхні, якщо контракт не підтримується. |
| **Accumulator / акумулятор** | Об'єкт, який накопичує результат злиття без створення повного нового списку після кожної дельти. |
| **Coordinator / координатор** | Компонент, який приймає завершені результати, буферизує їх і випускає в правильному порядку. |
| **`SortedDictionary`** | Колекція .NET, яка зберігає ключі у відсортованому порядку. У книзі вона використовується для буфера завершень за sequence. |
| **Red-Black Tree** | Самобалансоване дерево пошуку. `SortedDictionary` у .NET реалізований на такому типі структури. |
| **Out-of-order** | Ситуація, коли батч із більшим номером завершується раніше за батч із меншим номером. |
| **Contiguous range** | Неперервний діапазон sequence без прогалин. Координатор може випускати лише таку послідовність. |
| **Stale topology recompute** | Перерахунок результату, якщо батч був оброблений за старою версією топології. |

## Шарди, топологія і балансування

| Термін | Просте пояснення |
| :--- | :--- |
| **Partition / розділ** | Логічна частина роботи або даних. У книзі часто відповідає ідеї shard/partition у системах обробки потоків. |
| **Shard / шард** | Частина системи, яка відповідає за певний набір source-ів або розділів роботи. |
| **Hot partition** | Розділ, який отримує непропорційно багато роботи, наприклад через локальний шторм. |
| **Topology** | Карта розподілу source-ів по шардах і воркерах. |
| **Topology version** | Версія цієї карти. Потрібна, щоб відрізняти актуальні й застарілі обчислення. |
| **Rebalance / ребалансування** | Перерозподіл роботи між шардами, коли навантаження стало нерівним. |
| **Migration / міграція** | Перенесення source-а або групи роботи з одного шарда на інший. |
| **Overship** | Перенесення зайвої частки роботи під час міграції. У книзі це пояснюється як практичний ризик неідеального перенесення навантаження. |
| **Claim** | Заява права на роботу або ресурс. У топології та durable-контурі claim означає: “цей виконавець зараз відповідає за це”. |
| **Skew** | Перекіс розподілу: коли частина джерел або шардів отримує набагато більше навантаження за інші. |
| **Hysteresis / гістерезис** | Запобіжник проти метушні: система не змінює рішення одразу після дрібного коливання метрики. |
| **Cooldown / час охолодження** | Період, протягом якого система не дозволяє нову міграцію після попередньої. |
| **Migration budget / бюджет міграцій** | Ліміт на кількість або частоту переносів, щоб ребаланс не став джерелом хаосу. |

## Пам'ять, ownership і продуктивність

| Термін | Просте пояснення |
| :--- | :--- |
| **GC / Garbage Collector** | Збирач сміття .NET: автоматично звільняє пам'ять, але може створювати паузи й накладні витрати. |
| **Heap / купа** | Область пам'яті для об'єктів, якими керує GC. |
| **Allocation / алокація** | Виділення пам'яті під новий об'єкт або буфер. |
| **LOH / Large Object Heap** | Частина купи .NET для великих об'єктів. Часті великі алокації можуть дорого коштувати. |
| **Stop-the-World** | Пауза, коли GC зупиняє виконання потоків, щоб виконати частину роботи зі збирання пам'яті. |
| **`ArrayPool<T>`** | Пул масивів у .NET: дозволяє орендувати й повертати буфери замість постійного створення нових. |
| **Pooled-copy** | Копіювання payload у буфер, орендований з пулу. Це зменшує тиск на GC порівняно з постійними новими масивами. |
| **Snapshot-copy** | Копіювання зліпка даних у нову пам'ять. У книзі саме воно спричинило кризу утриманих алокацій. |
| **Owned payload** | Payload, життєвим циклом якого система явно володіє: знає, хто його тримає і коли його треба звільнити. |
| **Ownership protocol** | Правила володіння буфером: хто отримав, хто читає, хто повертає, хто відповідає за помилку. |
| **Retained payload** | Payload, який система ще утримує після первинної обробки. |
| **Retained pressure** | Тиск утриманої пам'яті: скільки payload-байтів досі не відпущено. |
| **Capacity evidence** | Доказ місткості: метрики, які показують, що система не лише швидка, а й тримає контроль над пам'яттю та чергами. |
| **Prewarm / прогрів** | Попередня підготовка пулів, JIT або буферів, щоб перший реальний прогін не платив повну ціну холодного старту. |
| **Cache locality / локальність кешу** | Властивість даних лежати в пам'яті так, щоб процесор міг читати їх послідовно й ефективно. |
| **Mechanical sympathy** | Інженерна увага до реальної фізики CPU, кешів, пам'яті й runtime, а не лише до абстракцій коду. |

## Асинхронність, черги і backpressure

| Термін | Просте пояснення |
| :--- | :--- |
| **Async transport** | Асинхронний транспорт між частинами системи: робота передається через чергу, а не прямим синхронним викликом. |
| **Channel** | .NET-механізм передачі даних між producer і consumer. |
| **Bounded channel** | Channel з обмеженою місткістю. Він не дозволяє системі накопичити нескінченну чергу. |
| **`BoundedChannelFullMode.Wait`** | Режим, у якому producer чекає, якщо bounded channel заповнений. Так backpressure стає явним. |
| **Backpressure** | Зворотний тиск: сигнал, що споживачі не встигають, тому виробники мають сповільнитися або чекати. |
| **Producer** | Частина системи, яка створює роботу або кладе її в чергу. |
| **Consumer** | Частина системи, яка забирає роботу з черги й обробляє її. |
| **Queue depth** | Поточна глибина черги. |
| **High-watermark** | Найбільше зафіксоване значення певної метрики, наприклад максимальної глибини черги. |
| **Drain** | Дренування черги або буфера: контрольований випуск накопичених елементів. |
| **Decoupling / розчеплення** | Розділення producer і consumer так, щоб вони не блокували одне одного прямим викликом. |

## Стійкість, durable-контур і відновлення

| Термін | Просте пояснення |
| :--- | :--- |
| **Durable** | Стійкий до перезапуску або збою. Дані не живуть лише в оперативній пам'яті. |
| **Durable envelope** | “Конверт” роботи зі станом, ідентичністю і переходами життєвого циклу. |
| **Envelope state** | Поточний стан durable envelope: наприклад claimed, completed, committed або blocked. |
| **FSM / скінченний автомат** | Finite State Machine: модель, де об'єкт може бути лише в одному з наперед визначених станів і переходити між ними за правилами. |
| **Poison batch** | Батч, який не вдається обробити коректно навіть після повторних спроб. |
| **Quarantine / карантин** | Ізоляція проблемної роботи, щоб вона не ламала весь потік. |
| **Retry limit** | Ліміт повторних спроб перед тим, як система визнає проблему сталою. |
| **Idempotency / ідемпотентність** | Повторне виконання тієї самої операції не змінює результат понад один раз. |
| **File durable store** | Файлове сховище durable-стану. У RadarPulse це перший простий адаптер довговічності, не заміна повноцінній базі або брокеру. |
| **Temp-file replacement** | Запис через тимчасовий файл із подальшою заміною основного файла. Це зменшує ризик частково записаного JSON-документа. |
| **`File.Move(..., overwrite: true)`** | .NET-виклик, який використовується для заміни файла історії/стану після підготовки тимчасового файла. |
| **Atomic file move** | Атомарне перейменування/переміщення файла, яке використовується для безпечнішого запису цілого JSON-документа. |
| **`fsync`** | Операція примусового скидання даних на диск. У книзі згадується як гарантія, яку поточний локальний файловий store не заявляє. |
| **WAL** | Write-Ahead Log: журнал попереднього запису. У книзі згадується як те, чим поточний файловий history store не є. |
| **Raft / Paxos** | Алгоритми консенсусу для розподілених систем. У книзі вони згадуються як production-рівень, який не реалізовано в поточному стенді. |

## Діагностика, логування і доказова інженерія

| Термін | Просте пояснення |
| :--- | :--- |
| **Evidence / доказ** | Те, що можна перевірити: код, тест, вимірювання, лог, віху або явно названу межу. |
| **Claim / твердження** | Заява про систему, яка має мати доказ і межу відповідальності. |
| **Milestone evidence** | Доказ із віхи розробки: зафіксоване рішення, тест, benchmark або closeout. |
| **Performance gate** | Перевірка, яка не дає прийняти зміну без збереження потрібного рівня продуктивності або пам'яті. |
| **Benchmark** | Вимірюваний сценарій продуктивності. У книзі benchmark завжди прив'язаний до корпусу, заліза й режиму запуску. |
| **Smoke test** | Швидка перевірка, що маршрут живий. Не дорівнює повному доказу продуктивності. |
| **Verify** | Команда demo-пакета, яка проходить збірку, тести, UI-smoke, API/readiness gates та інші перевірки. |
| **Readiness gate** | Перевірка, що потрібна поверхня готова й не має блокера. |
| **Architecture gate** | Перевірка архітектурних меж, наприклад заборонених залежностей між шарами. |
| **Diagnostic contract** | Типізована мова фактів, які система має сказати про себе: readiness, blockers, retained pressure, queue telemetry. |
| **Queue telemetry** | Підсумки роботи черги: кількість спроб enqueue, заповнення, timeout, high-watermark і recent details. |
| **Observability / спостережуваність** | Здатність пояснити стан системи через метрики, логи, траси й діагностичні моделі. |
| **Structured logging** | Логування не текстовим потоком “що сталося”, а структурованими подіями з полями. |
| **OpenTelemetry** | Стандарт і набір інструментів для метрик, логів і трас. У книзі це майбутній adapter layer над уже готовою діагностичною моделлю. |
| **Trace / траса** | Зв'язаний шлях операції через компоненти системи. |
| **Span** | Окрема ділянка траси: операція з початком, кінцем і атрибутами. |
| **Metric exporter** | Компонент, який виводить метрики в зовнішню систему моніторингу. |
| **Log sink / приймач логів** | Місце, куди потрапляють логи: файл, консоль, сервіс, pipeline. |
| **Correlation** | Зв'язування подій через `runId`, sequence, envelope id або інші поля. |
| **Incident workflow** | Процес роботи з інцидентом: виявити, локалізувати, відновити, пояснити, запобігти повторенню. |

| **Hardware boundary** | Межа твердження, прив'язана до конкретного заліза. Продуктивність без цієї межі легко перебільшити. |
| **Corpus boundary** | Межа твердження, прив'язана до конкретного набору вхідних даних. |
| **Release build** | Збірка з оптимізаціями, придатніша для performance-вимірювань, ніж Debug. |
| **Debug build** | Збірка для розробки й налагодження, яка не підходить як основа для performance-claim. |
| **Restore** | Відновлення NuGet/npm залежностей перед build/test. |
| **Browser smoke** | Швидка браузерна перевірка, яка підтверджує, що UI справді відкривається й взаємодіє з API. |
| **Nginx/YARP** | Зворотні проксі/шлюзи, які можуть знадобитися для production-масштабування HTTP-трафіку, але не входять у поточний локальний BFF-claim. |
| **Exit code** | Код завершення процесу. Ненульовий exit code у verify означає провал перевірки. |
