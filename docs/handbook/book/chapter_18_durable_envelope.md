# Розділ 18: Дипломатичний конверт ([`DurableEnvelope`](../../../src/Infrastructure/Processing/Durable/Services/RadarProcessingDurableEnvelopeQueue/RadarProcessingDurableEnvelopeQueue.cs))

У промислових архітектурах обробки великих даних для забезпечення надійності зазвичай використовують важку артилерію: Apache Kafka, RabbitMQ або розподілені черги AWS SQS. Вони дають готові примітиви довговічності (durability primitives), групи споживачів (consumer groups), підтвердження отримання повідомлень (acknowledgements) і контури повторних спроб та dead-letter-черг для повідомлень, які вже не можна безпечно обробляти автоматично.

Але ми будуємо наш «лабораторний стіл» (Lab Table) без зовнішніх залежностей. Ми відмовилися від ідеї розгортання складних брокерів черг, щоб зберегти відтворюваність тестів і швидкість локального процесу. Як же нам описати семантику доставки щонайменше один раз (At-Least-Once delivery) та виживання черги під час аварій, не перетворюючи книгу на інструкцію з експлуатації брокерів?

Розв'язанням цієї загадки під час віхи `023` стала розробка власного брокер-нейтрального контракту черги та концепції дипломатичного конверта — **DurableEnvelope**.

---

## 18.1. Паспорт дипломатичної пошти

Уявіть собі кур'єра, який перевозить цінні державні документи. Він не везе папери просто в руках. Вони запечатані в спеціальний шкіряний дипломатичний конверт із сургучною печаткою, унікальним номером та журналом передачі. Цей конверт не можна просто так викинути чи відкрити без реєстрації. Кожен його крок фіксується в протоколі безпеки.

У RadarPulse кожен вхідний батч радарних даних [`RadarEventBatch`](../../../src/Domain/Streaming/Batches/Models/RadarEventBatch.cs) під час надходження в систему запаковується в такий конверт — [`DurableEnvelope`](../../../src/Infrastructure/Processing/Durable/Services/RadarProcessingDurableEnvelopeQueue/RadarProcessingDurableEnvelopeQueue.cs). Конверт несе в собі не лише корисні дані, а й залізний набір метаданих для контролю життєвого циклу:

* **Стабільний ідентифікатор батча ([`RadarProcessingDurableBatchId`](../../../src/Domain/Processing/Durable/Models/RadarProcessingDurableBatchId.cs)):** Глобально унікальний маркер пакета, який не змінюється при перезапусках.
* **Порядковий номер провайдера ([`ProviderSequence`](../../../src/Domain/Processing/Queueing/Models/RadarProcessingQueuedBatchSequence.cs)):** Наша вісь часу, що визначає суворий порядок обробки та фіксації.
* **Номер спроби (`Attempt`):** Лічильник, що показує, скільки разів воркери намагалися обробити цей конверт.
* **Ідентифікатор воркера ([`WorkerId`](../../../src/Domain/Processing/Workers/Models/RadarProcessingWorkerId.cs)):** Маркер потоку чи процесу, який взяв конверт в оренду для обробки.
* **Часові мітки життєвого циклу:** Точні дати прийняття (`Accepted`), резервування (`Claimed`), завершення обчислень (`Completed`), фіксації (`Committed`) та вивільнення ресурсів (`Released`).

Сам конверт виступає в ролі скінченного автомата (Finite State Machine, FSM): моделі, де об'єкт у кожен момент перебуває лише в одному дозволеному стані й може переходити тільки визначеними правилами. Основний шлях — `Pending -> Claimed -> Completed -> Committed -> Released`, але аварійні стани мають власні переходи:

```text
                         Accept
                           |
                           v
                    +-------------+
                    | Pending     |
                    +------+------+
                           |
                       ClaimNext
                           |
                           v
                    +-------------+
                    | Claimed     |
                    +------+------+
                           |
             +-------------+-------------+
             |             |             |
          Complete       Fail         Abandon
             |             |             |
             v             v             v
      +-------------+ +-------------+ +-------------+
      | Completed   | | Failed      | | Abandoned   |
      +------+------+ +------+------+ +------+------+
             |             |             |
        MarkCommitted      |             |
             |             |             |
             v             |             |
      +-------------+      |             |
      | Committed   |      |             |
      +------+------+      |             |
             |             |             |
             |        +----+-------------+----+
             |        |                       |
             |      Retry                   Poison
             |        |                       |
             |        v                       v
             |  +-------------+         +-------------+
             |  | Pending     |         | Poison      |
             |  +-------------+         +------+------+
             |                               |
             +---------------+---------------+
                             |
                         MarkReleased
                             |
                             v
                       +-------------+
                       | Released    |
                       +-------------+

Cancel / CancelOpen:
  Pending, Claimed, Completed -> Canceled -> Released

MarkReleased також може звільнити закриті або ресурсно завершені стани:
  Completed, Committed, Failed, Poison, Abandoned, Canceled -> Released

Короткі аварійні переходи:
  Claimed, Completed -> Failed
  Claimed, Completed -> Poison через Fail(..., poison: true)
```

Цей автомат робить втрату повідомлення видимою в межах локального durable-контракту, тобто контракту, який зберігає стан повідомлення після аварійного сценарію. Конверт або проходить шлях до фіксації, або осядає в журналі збоїв як доказ для інженера-детектива; книга не перетворює це на заяву про розподілене exactly-once.

## 18.2. Розділення завершення та фіксації

Критично важливим кроком під час віхи `023` стало повне розділення двох понять: **завершення роботи воркера (Worker Completion)** та **зовнішньої фіксації (Externally Visible Commit)**.

Коли воркер закінчує розрахунки над конвертом №4, він переводить конверт у стан `Completed`. Але це ще не означає, що дані стали частиною загального стану системи. Воркер всього лише виконав свою локальну роботу.

Якщо конверт №2 все ще перебуває в стані `Claimed` (повільний воркер ще рахує шторм), то конверт №4, незважаючи на свій успішний статус `Completed`, терпляче чекає своєї черги в буфері. Він не може бути зафіксований, оскільки це порушило б послідовність провайдера.

Ця затримка називається «блокуванням незавершеним попередником». Завдяки цьому ми зберігаємо повну свободу паралельних воркерів на етапі обчислень, але фіксуємо зміни в ядрі строго за порядком.

## 18.3. Керування аваріями: скінченний автомат, ліміт спроб і карантин отрути

Що відбувається, коли розслідування злочину йде не за планом? Наприклад, під час обробки конверта виникає виняток (тимчасовий збій мережі або помилка доступу до файлу).

Тут вступає в дію політика повторних спроб — [`RadarProcessingDurableRetryPolicy`](../../../src/Domain/Processing/Durable/Models/RadarProcessingDurableRetryPolicy.cs). Замість того, щоб кидати процес і падати в паніці, система діє за суворим протоколом скінченного автомата.

### Анатомія станів скінченного автомата:
* **Pending (Очікування):** Повідомлення знаходиться в черзі й готове до обробки. Будь-який вільний воркер може отримати на нього права.
* **Claimed (Зарезервовано):** Воркер орендував повідомлення. Записується його [`WorkerId`](../../../src/Domain/Processing/Workers/Models/RadarProcessingWorkerId.cs) та мітка `ClaimedAt`. Поки діє резерв, інші воркери не бачать це повідомлення.
* **Completed (Обчислено):** Воркер успішно розрахував дельту батча. Зміни лежать у координаторі результатів, але ще не зафіксовані в основному сховищі.
* **Committed (Зафіксовано):** Результати послідовно записані в [`RadarProcessingCore`](../../../src/Domain/Processing/Core/Services/RadarProcessingCore/RadarProcessingCore.cs). Повідомлення вважається успішно опрацьованим.
* **Failed (Збій):** Під час обробки виникла помилка. Конверт фіксує опис помилки і готується до ретраю.
* **Abandoned (Покинуто):** Воркер, який орендував конверт, раптово «зник» (наприклад, потік завис або таймаут обробки вичерпано). Конверт вважається покинутим.
* **Poison (Отрута):** Конверт повністю вичерпав ліміт спроб і більше не підлягає обробці.

### Синхронізація переходів
Усі переходи між цими станами в пам'яті мають бути потокобезпечними. [`RadarProcessingDurableEnvelopeQueue`](../../../src/Infrastructure/Processing/Durable/Services/RadarProcessingDurableEnvelopeQueue/RadarProcessingDurableEnvelopeQueue.cs) робить це без екзотики: тримає внутрішній `sync` lock, перевіряє попередній стан конверта і лише тоді змінює його. Воркери можуть працювати паралельно, але скінченний автомат не дозволяє двом різним історіям одночасно переписати один і той самий конверт. Стан рухається тільки вздовж дозволених ребер графу переходів.

### Політика повторних спроб без театру
Поточний [`RadarProcessingDurableRetryPolicy`](../../../src/Domain/Processing/Durable/Models/RadarProcessingDurableRetryPolicy.cs) навмисно невеликий: `MaxAttempts`, `RetryFailed`, `RetryAbandoned`. Він відповідає на одне питання: чи можна ще раз повернути конкретний конверт у стані `Failed` або `Abandoned` до `Pending`, чи доказ уже треба ізолювати як `Poison`.

Це не планувальник (scheduler) з `NextAttemptAt`, не експоненційне збільшення паузи між спробами (exponential backoff) і не випадкове розсіювання затримок (jitter). Для локального однопроцесного стенда (single-node) такий вибір важливий: ми доводимо семантику переходів і відновлення (recovery), а не ховаємо проблему за автоматичними затримками. Коли з'явиться адаптер до брокера (broker adapter) або справжнє міжпроцесне середовище виконання (cross-process runtime), backoff із jitter стане природним наступним шаром. У цій главі ми не продаємо його як виконану роботу.

### Визнання отруйним (Poison) та блокування черги
Якщо ліміт спроб вичерпано (наприклад, після 3 невдалих запусків), конверт переходить у статус `Poison` (отруйний пакет). У поточному середовищі виконання такий конверт ізолюється від звичайної видачі воркерам і лишається доказом системного збою, доки оператор не прийме явне рішення щодо відновлення.

Поява отруєного або аварійного (`Failed`) конверта в черзі блокує фіксацію наступних конвертів. Система не може рухатися далі, оскільки в ланцюжку подій утворилася діра. Черга зупиняється, очікуючи на втручання оператора.

## 18.4. Зведений рапорт: Панель готовності оператора

Для того, щоб детектив міг швидко оцінити стан системи, ми розробили звіт про готовність середовища обробки — [`RadarProcessingDurableRuntimeReadinessSummary`](../../../src/Domain/Processing/Durable/Models/RadarProcessingDurableRuntimeReadinessSummary.cs).

Цей звіт зчитується в режимі реального часу і містить вичерпні метрики для діагностики:

* **Readiness (Boolean):** Чи готова система приймати нові дані. Стає `false`, якщо лишився незавершений або заблокований конверт, є помилка вивільнення ресурсів або конверти у фінальних станах все ще тримають корисне навантаження (payload) у пам'яті.
* **Oldest Uncommitted Provider Sequence:** Номер найдавнішого конверта, який ще не пройшов фазу фіксації.
* **First Blocking Envelope:** Ідентифікатор першого конверта, який зупинив рух усієї черги.
* **First Blocking State & Reason:** Точний стан (наприклад, `Poison` або `Abandoned`) та текстовий опис причини блокування.
* **Terminal Retained Envelope Count & Payload Bytes:** Кількість конвертів та обсяг оперативної пам'яті, які зараз утримуються в буферах. Після повного очищення черги ці показники мають обов'язково повернутися до нуля.

Впровадження [`DurableEnvelope`](../../../src/Infrastructure/Processing/Durable/Services/RadarProcessingDurableEnvelopeQueue/RadarProcessingDurableEnvelopeQueue.cs) дозволило створити надійний, відмовостійкий каркас обробника радарних даних прямо в оперативній пам'яті нашого лабораторного процесу, підготувавши його до наступного кроку — збереження на дисковий носій.

## 🔍 Матеріали справи (Investigation Case Files)

### 1. Вердикт детективів (Decision Trace & Rationale)
Створення брокер-нейтрального контракту [`DurableEnvelope`](../../../src/Infrastructure/Processing/Durable/Services/RadarProcessingDurableEnvelopeQueue/RadarProcessingDurableEnvelopeQueue.cs) (Віха `023`). Замість інтеграції з важкими чергами Kafka або RabbitMQ, система використовує легкий скінченний автомат конверта повідомлення із станами: `Pending`, `Claimed`, `Completed`, `Committed`, `Failed`, `Poison`, `Abandoned`, `Canceled`, `Released`.

#### Чому ми не почали з великого брокера
Kafka, RabbitMQ або хмарна черга (cloud queue) могли б дати готові механізми доставки, але тоді книга про рішення на рівні середовища виконання швидко стала б книгою про налаштування брокера. Інший шлях — тримати все в пам'яті й не переживати падіння процесу, але це не витримує аварійного сценарію.

[`DurableEnvelope`](../../../src/Infrastructure/Processing/Durable/Services/RadarProcessingDurableEnvelopeQueue/RadarProcessingDurableEnvelopeQueue.cs) став нейтральним контрактом: стан повідомлення описаний у домені, а спосіб зберігання можна замінити. Ціна вибору — власний скінченний автомат і тести переходів; виграш — брокер можна підключити пізніше, не переписуючи семантику черги.

### 2. Закони фізики рантайму (System Invariants)
* **Політика повторних спроб**: У разі помилки обробки конверт повертається в стан `Pending` згідно з `RetryPolicy`.
* **Карантин отрути**: При перевищенні ліміту спроб конверт переходить у стан `Poison` та вилучається з активної черги.

### 3. Патологоанатомічний звіт (Failure Modes & Recovery)
* **Постійний збій пакета**: Якщо батч викликає помилку (наприклад, невідомий формат даних), система не зависає, а ізолює його в статус `Poison`, зберігаючи стабільність решти конвеєра.
