# Додаток В: план підготовки до продукційної експлуатації

Цей додаток не стверджує, що RadarPulse вже є продукційною платформою. Його завдання інше: показати, як автор переносив би доведений лабораторний рантайм у продукційний контур, не ламаючи ті інваріанти, заради яких книга взагалі була написана.

Сильний продукційний план починається не з Kubernetes і не з Kafka. Він починається з питання: **яку властивість системи ми вже довели, і що саме може зламатися, коли ми винесемо її за межі одного процесу?**

## Вихідна межа

Поточна книга доводить локальний контур одного вузла:

```text
корпус архіву -> потоковий контракт -> утримане корисне навантаження -> обчислення дельт
-> впорядкована фіксація -> стійкий конверт/файлове сховище -> BFF-моделі читання
-> інтерфейс оператора -> протокол готовності демо
```

Це сильний контур для продуктивності, дисципліни пам'яті, коректності паралельності й мислення про режими відмов. Але він навмисно не претендує на:

* приймання даних наживо з радарної мережі;
* доставку між машинами;
* безпеку публічного API;
* сертифікацію стійкості брокера або бази даних;
* автоскейлінг, автоматизацію розгортання та маршрутизацію сповіщень;
* розподілену обробку "рівно один раз".

Підготовка до продукційної експлуатації починається там, де ці незаявлені твердження стають реальними вимогами продукту.

Цей додаток не є повним контрольним списком продукційного запуску. Він свідомо не розкриває окремо керування секретами, резервне копіювання і відновлення, відкат розгортань чи інструкції реагування на інциденти. Його фокус вужчий: як переносити вже доведені інваріанти RadarPulse в нові зовнішні залежності й не втрачати доказову дисципліну.

## Порядок введення продукційних шарів

| Крок | Що вводити | Чому саме тоді | Перша доказова перевірка |
| :--- | :--- | :--- | :--- |
| 1 | Контракт спостережуваності: структуровані логи, метрики, трасування, ідентифікатор запуску, ідентифікатор послідовності, версія топології, тиск утриманих ресурсів; починати з [Розділу 26](chapter_26_observability_logging.md) | Без цього продукційний збій не буде відтворюваним | Запуск із ін'єкцією відмов показує першу причину блокування, тиск утриманих ресурсів, шлях повтору або poison-повідомлення і ідентифікатор кореляції |
| 2 | Адаптер бази даних для історії запусків, готовності або стійкого стану | Коли локальний JSON перестає бути достатнім для аудиту, утримання історії або доступу кількох користувачів | Контрактні тести адаптера: версія схеми, межа транзакції, повтор дубліката, обробка пошкодженого запису |
| 3 | Адаптер брокера для [`DurableEnvelope`](../../../src/Infrastructure/Processing/Durable/Services/RadarProcessingDurableEnvelopeQueue/RadarProcessingDurableEnvelopeQueue.cs) | Коли життєвий цикл виробника й воркера виходить за межі одного процесу | Тести повторного програвання щонайменше один раз, дубльована доставка, карантин poison-повідомлень, впорядкована фіксація після перезапуску |
| 4 | Межа публічного API: зворотний проксі, завершення TLS, автентифікація, авторизація, обмеження частоти запитів, політика CORS | Коли інтерфейс оператора перестає бути локальною панеллю керування | Швидка базова перевірка безпеки API, матриця прав, відхилення небезпечних команд керування |
| 5 | Адаптер приймання даних наживо | Коли система має працювати з реальним зовнішнім потоком, а не корпусом повторного програвання | Зворотний тиск, пошкоджений запис, перепідключення, дрейф корпусу й перевірки бюджету затримки |
| 6 | Обробка на кількох вузлах | Тільки після брокера, бази даних і спостережуваності, бо інакше немає захисного відсікання старого власника й історії відновлення | Право власності на розділ, маркер права власності в топології, паритет перерахунку застарілого маршруту, сценарій примусового завершення вузла |

Цей порядок навмисно консервативний. Він не максимізує кількість технологій у діаграмі; він мінімізує шанс втратити вже доведену коректність.

Poison-повідомленням тут називається повідомлення, яке не вдалося коректно обробити після дозволеної кількості повторів. Його не можна просто проковтнути або безкінечно повертати в чергу: система має ізолювати його, показати причину й не псувати впорядковану фіксацію решти потоку.

У рядку про публічний API важливо розділяти дві різні речі. **Автентифікація** відповідає на питання: хто робить запит? **Авторизація** відповідає на інше питання: чи має цей користувач або сервіс право виконати саме цю дію? Для RadarPulse це не формальність навколо екрана входу. Особливо небезпечні команди керування — зупинка, drain (дочекатися спорожнення черг), cancel (скасування), recovery (відновлення) — мають бути дозволені явно, а не лише заховані за фактом логіну.

Для обробки на кількох вузлах так само важливе **захисне відсікання старого власника**. Якщо вузол A володів розділом, потім втратив маркер права власності або застарів за версією топології, а вузол B отримав нове право власності, пізня фіксація від вузла A має бути відхилена. Інакше система отримає роздвоєння власності: два процеси вважають себе власниками одного розділу і можуть записати несумісну історію. Практично це зазвичай робиться через монотонний маркер права, версії або токена, який перевіряється перед фіксацією.

## Чого не робити першим

Найпростіший спосіб зробити RadarPulse одночасно більшим на вигляд і слабшим по суті:

* перенести поточний процес у Kubernetes до визначення SLO, тобто цілей рівня сервісу;
* замінити локальне файлове сховище на Kafka без збереження семантики конверта;
* додати WebSocket-візуалізацію до визначення візуальних DTO і браузерних перевірок продуктивності;
* додати екрани входу без авторизаційної семантики для небезпечних команд керування;
* назвати адаптер брокера або бази даних готовим до продукційної експлуатації після самих тестів щасливого шляху.

Книга здобуває довіру саме тому, що не йде цими короткими шляхами. Продукційна версія має зберегти цю дисципліну.

## Сигнал для захисту

Цей план навмисно конкретний, бо зріла продукційна робота не зводиться до “додати хмару”. Це дисципліна перенесення доведених інваріантів через нові режими відмови. Автор уже показав лабораторну версію цієї дисципліни. Додаток показує наступну розмову на захисті: що зміцнювати першим, що відкласти і які докази потрібні, перш ніж продукційне твердження стане чесним.
