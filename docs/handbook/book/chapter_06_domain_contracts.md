# Розділ 6: Залізні контракти домену

Уявіть себе головою митної служби на кордоні суверенної держави. Щодня до вас на пропускний пункт прибувають тисячі вантажівок. Деякі везуть задекларовані товари, а деякі — контрабанду, пошкоджені продукти або небезпечні речовини. Якщо ваші прикордонники будуть ліниво махати рукою кожній машині, не заглядаючи в кузов і не перевіряючи паспорти водіїв, дуже швидко в країні почнеться хаос, спалахнуть епідемії, а ринки заповнить фальсифікат.

У системі **RadarPulse** кожен вхідний радарний батч даних — це такий вантажний автомобіль, який прибуває на кордон нашого доменного монастиря. Домен не має права довіряти зовнішньому світу на слово. Ми не можемо дозволити пошкодженим, неповним або сфальсифікованим даним проникнути вглиб обчислювального ядра. Будь-яка помилка в розмітці байтів або порушення хронології зруйнує логіку обробки та перетворить фінальну метеорологічну карту на набір безглуздого шуму.

Тому під час Віхи `035` (Code Contract Documentation Pass) ми зафіксували та детально описали залізні митні правила нашої системи — контракти валідації в класі `RadarEventBatchValidator`.

## 6.1. Митний догляд: 8 рівнів перевірки `RadarEventBatch`

Коли новий батч `RadarEventBatch` потрапляє в систему, він негайно передається в руки доменного інспектора — статичного класу `RadarEventBatchValidator`. Цей інспектор не знає пощади й перевіряє батч за вісьмома суворими критеріями.

### 1. Версія схеми потоку (Stream Schema Version)
Перше, що робить митник — перевіряє версію паспорта батча. Якщо версія схеми даних батча не збігається з поточною системною версією (`StreamSchemaVersion.Current`), батч негайно відхиляється. Ми не працюємо з застарілими чи невідомими форматами документів:
```csharp
if (batch.StreamSchemaVersion != StreamSchemaVersion.Current)
{
    return Invalid(RadarEventBatchValidationError.UnsupportedStreamSchemaVersion, ...);
}
```

### 2. Паспорт джерела (Source Universe Version Mismatch)
Батч заявляє, що він зібраний для певної карти джерел (Source Universe). Якщо версія карти батча не збігається з реальною картою джерел, яку використовує наше обчислювальне ядро, це означає, що водій приїхав з картою іншого міста. Пропуск закривається:
```csharp
if (batch.SourceUniverseVersion != sourceUniverse.Version)
{
    return Invalid(RadarEventBatchValidationError.SourceUniverseVersionMismatch, ...);
}
```

### 3. Версія каталогу (Dictionary Snapshot Version Mismatch)
Якщо в батчі використовуються стиснуті коди радарів та типів фізичних величин (моментів), вони повинні розкодовуватися за допомогою словника (Dictionary Snapshot). Митник перевіряє, чи збігаються версії словника батча та словника системи, запобігаючи плутанині в термінах.

### 4. Хронологічний порядок (Chronology Order Violation)
Час не може йти назад. Події всередині батча повинні йти строго за неспадним (монотонно зростальним) порядком часу повідомлення (`MessageTimestampUtcTicks`). Якщо хоч одна подія має мітку часу меншу, ніж попередня, ми фіксуємо порушення лінії часу:
```csharp
if (streamEvent.MessageTimestampUtcTicks < previousMessageTimestampUtcTicks)
{
    return Invalid(RadarEventBatchValidationError.ChronologyOrderViolation, ...);
}
```

### 5. Фізичні розміри корисного навантаження (Payload Length Mismatch)
Для кожної події ми вираховуємо очікувану довжину корисного навантаження на основі кількості точок (gates) та ширини кожної точки в байтах (word size). Якщо реальна довжина байтового сегмента не збігається з розрахунковою — це означає, що частина даних відрізана або сфальсифікована.

### 6. Цілісність кордонів пам'яті (Payload Reference Outside Storage)
Митник перевіряє, чи не намагається подія вказати на зміщення в пам'яті, яке лежить за межами виділеного батчу байтового буфера. Це критично важлива перевірка безпеки (Security Gate), яка запобігає атакам типу «out-of-bounds read», коли зловмисник намагається прочитати чужі дані з оперативної пам'яті системи.

### 7. Неперервність корисного навантаження (Contiguous Payload References)
Байтові дані подій у спільному буфері повинні лежати «стик-в-стик» (contiguous), без порожніх дірок та без накладання подій одна на одну. Будь-який пропуск або перекриття пам'яті вважається ознакою пошкодження структури батча. Якщо після перевірки всіх подій у хвості буфера залишаються невідомі байтові сироти (`PayloadTailNotReferenced`), батч маркується як пошкоджений.

### 8. Верифікація джерел (Source Key & ID Validation)
Ми перевіряємо фізичні координати джерела події (код локатора, висота, азимут, смуга дальності). Ці координати повинні існувати в нашому всесвіті джерел (`RadarSourceUniverse`), а отриманий на основі цих координат унікальний числовий ідентифікатор `SourceId` повинен точно збігатися з тим, що записаний у події.

## 6.2. Філософія Fail-Closed: Краще зупинитися, ніж збрехати

Якщо батч не проходить хоча б одну з цих перевірок, система RadarPulse діє за жорстким правилом: **Fail-Closed** (режим безпечного закриття).

Ми не намагаємося «виправити» пошкоджені дані на ходу. Ми не відкидаємо «погані» події, сподіваючись обробити хоча б «хороші». Чому? Тому що в радарних обчисленнях напівправда гірша за мовчання. Якщо метеоролог отримає спотворені дані, де переплутані координати або пропущені байти, він може пропустити зародження смертоносного урагану.

При першому ж виявленні порушення контракту:
1. `RadarEventBatchValidator` повертає структурований об'єкт помилки `RadarEventBatchValidationResult` із докладним описом порушення, індексом зламаної події та очікуваними значеннями.
2. Обчислювальне ядро негайно припиняє обробку всієї поточної сесії.
3. Тимчасові буфери воркерів очищуються, виділена пам'ять повертається в пули.
4. Помилка записується в журнал і виводиться на екран оператора в Angular UI з червоною позначкою тривоги.

Це гарантує абсолютну надійність. Оператор точно знає: якщо на екрані світиться зелений маркер успішного комміту, дані пройшли найсуворіший контроль якості на кордоні нашого доменного монастиря, і результатам аналізу можна довіряти на всі сто відсотків.

## 🔍 Матеріали справи (Investigation Case Files)

### 1. Вердикт детективів (Decision Trace & Rationale)
Формалізація залізних контрактів домену (Milestone `035`). Обробка та валідація вхідних даних здійснюються через чіткі інтерфейси, такі як `IRadarProcessingCore`. Це усунуло розмитість логіки та гарантувало, що некоректно сформовані запити відхиляються на самому кордоні домену, не забруднюючи внутрішню пам'ять ядра.

### 2. Закони фізики рантайму (System Invariants)
* **Валідація при конструюванні**: Об'єкти домену не можуть перебувати в невалідному стані — будь-яке створення сутності супроводжується миттєвою перевіркою параметрів.
* **Незмінність (Immutability)**: Результати обробки домену є незмінними структурами.

### 3. Патологоанатомічний звіт (Failure Modes & Recovery)
* **Контрактний збій**: При передачі некоректних даних (наприклад, від'ємного часу або невідомого коду радара) система миттєво викидає `DomainValidationException` та перериває обробку батча.
